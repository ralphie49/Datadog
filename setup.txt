DATADOG ANALYSER
=============================================

1) FOLDER STRUCTURE
--------------------
Create a project folder (name it anything you like — nothing in the pipeline hardcodes a
project name). Inside it, create exactly these three sibling folders:

your-project-folder/
├── agents/
│   ├── datadog-analyser-orchestrator.md
│   ├── log-ingestion-normaliser-agent.md
│   ├── error-data-quality-agent.md
│   ├── performance-infrastructure-health-agent.md
│   ├── pipeline-health-monitor-agent.md
│   ├── security-audit-agent.md
│   ├── anomaly-detection-agent.md
│   ├── dependency-flow-analysis-agent.md
│   ├── root-cause-analysis-agent.md
│   ├── code-patch-generator-agent.md
│   └── report-generation-agent.md
│
├── input/
│   └── (drop your Datadog export files here — any filenames,
│        any mix of .json / .csv / .log / .ndjson. The pipeline
│        auto-detects logs / metrics / traces / alerts /
│        infrastructure by content, not by filename.)
│
└── output/
    └── (starts empty — the pipeline writes all 10 result files
         here when it runs: normalised_data.json, log_analysis.json,
         metrics_report.json, apm_report.json, security_report.json,
         anomaly_report.json, dependency_report.json, root_cause.json,
         recommendations.json, patch_suggestions.json, and the final
         datadog_analysis_report.md)


2) HOW TO RUN IT
-----------------------------------------------------------------
PROMPT:

Run the **Datadog Analyser Orchestrator**: read `datadog-analyser-orchestrator.md`,
then follow each of the 10 agent `.md` files it references, in order, phase by
phase — obeying every MUST/MUST NOT rule literally. Use my export files in
`input/`, write each phase's output to `output/` with the exact filenames
specified, and finish by generating `output/datadog_analysis_report.md`.
-----------------------------------------------------------------


3) WHAT SHOULD COME OUT
--------------------------
After a full run, output/ should contain 11 files:

  normalised_data.json       (Phase 0 — ingestion)
  log_analysis.json          (Phase 1 — errors + data quality)
  metrics_report.json        (Phase 2 — latency/throughput/host health)
  apm_report.json            (Phase 3 — Kafka lag/checkpoints/SLA)
  security_report.json       (Phase 4 — PII/credential/auth findings)
  anomaly_report.json        (Phase 5 — spikes/drops/trends)
  dependency_report.json     (Phase 6 — service graph + breakpoints)
  root_cause.json            (Phase 7a — correlated incidents)
  recommendations.json       (Phase 7b — ranked fixes)
  patch_suggestions.json     (Phase 8 — draft diffs, human review only)
  datadog_analysis_report.md (Phase 9 — final consolidated report)

If any of these are missing, check the assistant's run log for which phase
stopped and why — the agent specs are written to fail loudly (stop + report)
rather than silently skip a phase.
